# سامانه CRM ماشین‌آلات راهسازی

سامانه اختصاصی مدیریت ارتباط با مشتری (CRM) برای صنعت ماشین‌آلات راهسازی، پیاده‌سازی‌شده با
Django 5 و PostgreSQL؛ شامل گردش کار کامل کسب‌وکار (تماس مشتری → ارجاع به تکنسین → برآورد
قیمت → تایید و پیش‌فاکتور → قرارداد → درخواست قطعه از انبار)، داشبورد نقش‌محور، تقویم شمسی،
پرتال اختصاصی مشتری، اعلان‌ها و پیام‌رسانی داخلی.

|---|---|---|

end-to-end گردش کار کسب‌وکار برای هر ۷ نقش اعتبارسنجی شده است.

## 🔄 گردش کار کسب‌وکار (دقیقاً مطابق سناریوی واقعی)

1. **مشتری تماس می‌گیرد** (مثلاً «برق لودرم خراب است») → مدیر فروش یا کارشناس فروش یک
   «درخواست تعمیر» (`ServiceRequest`) برای او ثبت می‌کند.
2. **مدیر فروش دستگاه را به تکنسین ارجاع می‌دهد** → از صفحه جزئیات درخواست، دکمه «تخصیص
   تکنسین» را می‌زند (`TaskAssignment`) → تکنسین بلافاصله اعلان دریافت می‌کند.
3. **تکنسین برآورد قیمت ثبت می‌کند** (`CostEstimate`: هزینه، ساعت کار، قطعات موردنیاز) →
   مدیر فروش اعلان دریافت می‌کند.
4. **مدیر فروش برآورد را تایید می‌کند** → به‌صورت خودکار یک «فرصت فروش» و «پیش‌فاکتور»
   ساخته می‌شود؛ مدیر فروش آن را نهایی/ارسال می‌کند (مشتری از طریق پرتال خودش هم می‌تواند
   ببیندش).
5. **قرارداد بسته می‌شود**: از صفحه پیش‌فاکتور، دکمه «ایجاد قرارداد از این پیش‌فاکتور» مبلغ
   را به‌صورت خودکار پر می‌کند.
6. **تکنسین کار را شروع می‌کند** و در صورت نیاز، از صفحه کار خودش «درخواست قطعه» می‌دهد
   (`PartRequest`) → انباردار اعلان دریافت می‌کند و با یک کلیک «تامین» می‌کند (که به‌صورت
   خودکار یک تراکنش خروج انبار هم ثبت می‌شود).
7. **مدیرعامل** از داشبورد خودش، کل روند (فرصت‌های باز، برآوردهای در انتظار تایید،
   درخواست‌های قطعه، وضعیت تعمیرات، نمودار فروش ماهیانه) را زیر نظر دارد.

## 🔐 دسترسی نقش‌محور در منو و داشبورد

هرکسی فقط همان بخش‌هایی از سایدبار/داشبورد را می‌بیند که به کارش مربوط است (بر پایه
Django Groups/Permissions که در `setup_groups` تعریف شده‌اند، نه فقط یک لیست ثابت در قالب):

- **تکنسین**: فقط «کارهای من»، درخواست‌های تعمیر تخصیص‌یافته به خودش، ثبت برآورد/گزارش
  تعمیر، درخواست قطعه از انبار. هیچ قیمت/قرارداد/نموداری نمی‌بیند.
- **انباردار**: فقط قطعات، تراکنش‌های انبار، درخواست‌های قطعه در انتظار. هیچ اطلاعات مالی
  فروش نمی‌بیند.
- **حسابداری**: فقط مشتریان، پیش‌فاکتورها، قراردادها (بدون امکان ویرایش). هیچ‌چیز از
  تعمیرگاه/انبار نمی‌بیند.
- **کارشناس فروش**: مشتریان، فرصت‌های خودش، پیش‌فاکتور، ثبت درخواست تعمیر برای مشتری.
- **مدیر فروش**: همه‌چیز مربوط به فروش + ارجاع دستگاه به تکنسین + تایید برآورد قیمت.
- **مدیر تعمیرگاه**: همه‌چیز مربوط به تعمیرگاه (تکنسین‌ها، مأموریت‌ها، درخواست‌های تعمیر).
- **مدیرعامل**: دسترسی کامل به همه‌چیز + همه نمودارها.

## 👤 پرتال مشتری

مدیر/کارشناس فروش می‌تواند از صفحه جزئیات هر مشتری (دکمه «ایجاد حساب پرتال برای مشتری»،
هم در پنل ادمین با اکشن گروهی و هم مستقیم در برنامه) یک حساب کاربری برای آن مشتری بسازد.
مشتری با این حساب وارد `/portal/` می‌شود (نه پنل داخلی) و می‌تواند:

- تاریخچه کامل درخواست‌های تعمیر خودش و روند دقیق هر تعمیر (تکنسین مسئول، گزارش‌های
  تعمیر ثبت‌شده) را ببیند.
- درخواست تعمیر جدید ثبت کند (همان کاری که تلفنی هم انجام می‌داد).
- پیش‌فاکتورها و قراردادهای خودش را مشاهده کند.

یک میان‌افزار (`CustomerPortalAccessMiddleware`) تضمین می‌کند این حساب‌ها به هیچ صفحه
مدیریتی داخلی (از جمله پنل ادمین) دسترسی نداشته باشند.

## 📅 تقویم شمسی

همه تاریخ‌ها در رابط کاربری به‌صورت شمسی نمایش داده و دریافت می‌شوند (مثلاً `1403/04/15`)؛
در پس‌زمینه و دیتابیس همچنان میلادی ذخیره می‌شوند (استاندارد و قابل‌حمل). این تبدیل با
کتابخانه `jdatetime` و یک فیلد فرم سفارشی (`apps/core_utils/fields.py`) انجام می‌شود.
کاربر می‌تواند تاریخ را مستقیم به‌صورت `1403/04/15` تایپ کند؛ افزودن یک تقویم گرافیکی
جاوااسکریپتی (مثل persian-datepicker) به‌عنوان تکمیل تجربه کاربری در آینده قابل‌اضافه‌شدن
است ولی برای کارکرد صحیح ضروری نیست.

## 🔔 اعلان‌ها، ⚙️ تنظیمات، 💬 پیام‌رسانی داخلی

- **اعلان‌ها**: زنگوله بالای صفحه با شمارنده خوانده‌نشده‌ها (هر ۳۰ ثانیه به‌روزرسانی
  می‌شود). برای رویدادهایی مثل تخصیص کار جدید، برآورد در انتظار تایید، درخواست قطعه، و
  غیره به‌صورت خودکار ساخته می‌شوند (`apps/notifications`).
- **تنظیمات**: منویی ساده برای ویرایش پروفایل و تغییر رمز، بدون نیاز به ورود به پنل ادمین
  (`apps/users` + `/accounts/settings/`).
- **پیام‌رسانی داخلی**: گفتگوی دو نفره ساده بین کارکنان برای هماهنگی سریع
  (`apps/chat`، نسخه اول بدون Real-time/WebSocket).
- **پروفایل کاربر**: هر کاربر می‌تواند عکس، نام، ایمیل و توضیح کوتاه خودش را از
  `/accounts/profile/` ویرایش کند؛ نام و سمت او در نوار بالای همه صفحات نمایش داده می‌شود.
- **آیکن چشم روی رمز عبور**: در تمام فرم‌های رمز عبور (ورود، تغییر رمز)، یک دکمه 👁 برای
  نمایش/مخفی‌کردن رمز اضافه شده (`static/js/app.js`).

## 🏗️ معماری

- **Backend**: Python 3.11 + Django 5
- **Database**: PostgreSQL 16
- **Task Queue**: Celery + Redis (هشدار سررسید چک‌ها، هشدار کمبود موجودی)
- **Frontend**: Bootstrap 5 (RTL) + فونت Vazirmatn
- **Auth**: Session-based با نقش‌های سفارشی (Django Groups/Permissions) و Rate Limiting
- **Auditing**: django-auditlog روی مدل‌های حساس (مشتری، فرصت، پیش‌فاکتور، قرارداد، چک، ...)

## ⚙️ نصب و راه‌اندازی (محیط توسعه)

### پیش‌نیازها
- Python 3.11+
- PostgreSQL 16 (یا Docker)
- Redis (یا Docker)

### روش ۱: نصب دستی

```bash
# ۱. ایجاد محیط مجازی
python3.11 -m venv venv
source venv/bin/activate  # ویندوز: venv\Scripts\activate

# ۲. نصب وابستگی‌ها
pip install -r requirements.txt

# ۳. تنظیم متغیرهای محیطی
cp .env.example .env
# فایل .env را ویرایش کرده و SECRET_KEY و DATABASE_URL را متناسب با محیط خود تنظیم کنید

# ۴. ایجاد دیتابیس در PostgreSQL
createdb crm_db   # یا از طریق psql: CREATE DATABASE crm_db;

# ۵. اجرای مایگریشن‌ها
python manage.py makemigrations
python manage.py migrate

# ۶. ساخت داده‌های نمونه (شامل کاربران، گروه‌های دسترسی، مشتریان، قراردادها)
python seed.py

# ۷. اجرای سرور توسعه
python manage.py runserver
```

سپس به آدرس `http://127.0.0.1:8000/accounts/login/` مراجعه کنید. کاربر ادمین پیش‌فرض:
`admin / AdminPass123!` (حتماً پس از اولین ورود رمز را تغییر دهید).

`seed.py` همچنین یک کاربر نمونه برای هر نقش می‌سازد (رمز همه: `Passw0rd!`) تا بتوانید تفاوت
منو/داشبورد هر نقش را ببینید:

| نام کاربری | نقش |
|---|---|
| `ceo1` | مدیرعامل (دسترسی کامل) |
| `salesmgr1` | مدیر فروش |
| `salesrep1` | کارشناس فروش |
| `workshopmgr1` | مدیر تعمیرگاه |
| `tech1` | تکنسین |
| `store1` | انباردار |
| `accountant1` | حسابداری |

```bash
cp .env.example .env
docker compose up --build
# در ترمینال دیگر:
docker compose exec web python manage.py migrate
docker compose exec web python seed.py
```

## 👥 نقش‌ها و دسترسی‌ها

نقش‌ها با دستور زیر به‌صورت گروه‌های Django با مجوزهای پیش‌فرض ایجاد می‌شوند
(این دستور به‌صورت خودکار توسط `seed.py` نیز اجرا می‌شود):

```bash
python manage.py setup_groups
```

نقش‌های موجود: مدیرعامل، مدیر فروش، کارشناس فروش، مدیر تعمیرگاه، تکنسین، انباردار، حسابداری.
جزئیات مجوز هر نقش در `apps/users/management/commands/setup_groups.py` قابل ویرایش است.

## 🧰 خلاصه امکانات هر ماژول

**تعمیرگاه (`apps/workshop`)**: لیست/فیلتر درخواست‌های تعمیر (تکنسین فقط کارهای خودش را می‌بیند)،
ثبت/ویرایش درخواست همراه با آپلود چند تصویر (Formset)، تخصیص تکنسین، ثبت گزارش روزانه تعمیر،
مدیریت لیست تکنسین‌ها، و ثبت/ویرایش مأموریت‌های خارج از تعمیرگاه.

**انبار (`apps/inventory`)**: لیست/فیلتر قطعات (جستجو + فیلتر «فقط کمتر از حداقل»)، CRUD کامل قطعات،
ثبت تراکنش ورود/خروج (با هشدار فوری در پیام سیستم و ایمیل Celery غیرهمزمان در صورت کمبود موجودی)،
و گزارش گردش کالا با فیلتر بازه زمانی/قطعه + خروجی CSV.

**گزارش‌گیری (`apps/reports`)**: داشبورد مدیریتی با ۴ نمودار Chart.js (فروش ماهیانه، وضعیت تعمیرات،
پربازده‌ترین تکنسین‌ها بر اساس ساعت کار، پرمصرف‌ترین قطعات) و خروجی گزارش خلاصه فروش/قراردادها در
سه فرمت PDF (`xhtml2pdf`)، Excel (`openpyxl`) و CSV.

## ⏱️ توضیح بخش «Periodic Tasks» در پنل ادمین

اگر با کاربر ادمین (`is_superuser`) وارد `/admin/` شوید، بخشی به نام **Periodic Tasks**
می‌بینید که متعلق به کتابخانه `django-celery-beat` است (برای زمان‌بندی خودکار تسک‌های
Celery، مثلاً بررسی دوره‌ای سررسید چک‌ها یا موجودی انبار در آینده). کاربران عادی (غیر
مدیرعامل) اصلاً به پنل ادمین دسترسی ندارند و نیازی به دانستن این بخش نیست؛ فقط برای
توسعه‌دهنده/مدیر فنی سیستم کاربرد دارد:

| مدل | کاربرد |
|---|---|
| **Periodic tasks** | لیست اصلی تسک‌های زمان‌بندی‌شده؛ هرکدام به یکی از سه نوع زمان‌بندی زیر (Interval/Crontab/Clocked/Solar) وصل می‌شوند و مشخص می‌کنند «کدام تابع Celery» و «چه زمانی» اجرا شود. |
| **Intervals** | زمان‌بندی ساده «هر X ثانیه/دقیقه/ساعت» (مثلاً «هر ۱۰ دقیقه یک‌بار»). |
| **Crontabs** | زمان‌بندی به سبک Cron لینوکس («هر روز ساعت ۸ صبح»، «هر دوشنبه»، و غیره) — برای گزارش‌های دوره‌ای یا هشدارهای زمان‌بندی‌شده مناسب است. |
| **Clocked** | اجرای یک‌باره در یک تاریخ/ساعت دقیق مشخص (نه تکرارشونده). |
| **Solar events** | زمان‌بندی بر اساس رویدادهای خورشیدی (طلوع/غروب آفتاب) بر مبنای مختصات جغرافیایی — برای این پروژه کاربردی ندارد و می‌توانید نادیده بگیرید. |

در حال حاضر پروژه از Celery فقط برای **کارهای فوری غیرهمزمان** (نه زمان‌بندی‌شده) استفاده
می‌کند: مثلاً ارسال ایمیل هشدار کمبود موجودی بلافاصله بعد از ثبت یک تراکنش خروج. اگر در
آینده خواستید کاری «هر روز صبح» یا «هر هفته» به‌صورت خودکار اجرا شود (مثلاً یادآوری چک‌های
نزدیک به سررسید)، دقیقاً از همین بخش Periodic Tasks استفاده خواهید کرد — کافی است یک
Crontab بسازید و آن را به تسک موردنظر وصل کنید، بدون نیاز به تغییر کد.

## 🆕 آخرین اضافات (بر اساس بازخورد کارفرما)

- **الزامی بودن عکس/فیلم در ثبت درخواست تعمیر از پرتال**: مشتری هنگام ثبت درخواست تعمیر
  باید حداقل یک عکس یا فیلم از خرابی/دستگاه ضمیمه کند (فرمت‌های مجاز: jpg، png، mp4، mov و
  مشابه)؛ بدون آن، فرم ذخیره نمی‌شود. این رسانه‌ها هم در پرتال مشتری و هم در صفحه داخلی
  درخواست تعمیر (برای تکنسین/مدیر تعمیرگاه) قابل مشاهده‌اند.
- **گردش کار تایید و ارسال پیش‌فاکتور**: کارشناس/مدیر فروش با دکمه «ارسال به مشتری برای
  تایید» پیش‌فاکتور را می‌فرستد (مشتری اعلان می‌گیرد) → مشتری از پرتال خودش «تایید» یا «رد»
  می‌کند (کارشناس فروش اعلان می‌گیرد) → فقط پس از تایید مشتری، دکمه «ایجاد قرارداد» در
  صفحه پیش‌فاکتور فعال می‌شود.
- **گردش کار تایید قرارداد توسط مشتری**: مشتری از پرتال خودش (صفحه جدید جزئیات قرارداد)
  می‌تواند قرارداد را تایید کند؛ با این تایید، مدیر تعمیرگاه و کارشناس فروش اعلان می‌گیرند
  که «کار تعمیر می‌تواند شروع شود». صفحه داخلی قرارداد هم وضعیت تایید مشتری را با یک
  نشان (badge) و هشدار نمایش می‌دهد (این یک راهنمای بصری است، نه قفل فنی سخت‌گیرانه روی
  ثبت گزارش تعمیر — چون مدل درخواست تعمیر و قرارداد به‌صورت مستقل طراحی شده‌اند تا
  انعطاف‌پذیری برای موارد استثنا هم حفظ شود؛ در صورت نیاز به قفل کامل، قابل‌اضافه‌شدن است).
- **رضایت‌سنجی ستاره‌ای پس از تحویل**: وقتی وضعیت درخواست تعمیر به «تحویل داده‌شده» برسد،
  در داشبورد و صفحه جزئیات پرتال مشتری یک فرم امتیازدهی ۱ تا ۵ ستاره به‌طور برجسته (بنر
  هشدار) نمایش داده می‌شود تا مشتری نظرش را ثبت کند. میانگین رضایت مشتریان هم به‌صورت یک
  کارت KPI در داشبورد مدیرعامل/مدیر تعمیرگاه اضافه شده است.

## 📝 نکات برای ادامه توسعه (پیشنهادی، نه ضروری)

- افزودن تقویم گرافیکی جاوااسکریپتی (مثل persian-datepicker) روی فیلدهای تاریخ برای تجربه
  کاربری بهتر (در حال حاضر تایپ مستقیم تاریخ شمسی به‌صورت `1403/04/15` کار می‌کند)
- افزودن هشدار خودکار Celery Beat برای سررسید چک‌ها (مدل `Check` آماده است؛ کافی است یک تسک
  دوره‌ای اضافه شود که چک‌های نزدیک به سررسید را بررسی و ایمیل ارسال کند — دقیقاً از طریق
  همان بخش Periodic Tasks که در بالا توضیح داده شد)
- ارتقای پیام‌رسانی داخلی به نسخه Real-time با Django Channels/WebSocket (نسخه فعلی ساده
  و مبتنی بر بارگذاری صفحه است)
- افزودن رمزنگاری اختیاری برای فایل‌های آپلودی حساس (طبق الزامات امنیتی پروپوزال)
- افزودن API (Django REST Framework) در صورت نیاز به اپلیکیشن موبایل یا یکپارچه‌سازی خارجی

## 🚀 نکات دیپلوی روی پروداکشن

پیش از دیپلوی، حتماً `DJANGO_SETTINGS_MODULE=core.settings.production`، `DEBUG=False`،
یک `SECRET_KEY` تصادفی و قوی، و HTTPS پشت Nginx را تنظیم کنید. برای فشرده‌سازی و کش فایل‌های
استاتیک از `python manage.py collectstatic` استفاده کنید (پیکربندی WhiteNoise از قبل انجام
شده است). سرویس‌های Gunicorn (وب) و Celery worker باید به‌صورت جداگانه (مثلاً با systemd یا
Supervisor) در پس‌زمینه اجرا شوند.
